For apprentices · Internal only

How to ship your first week.

You're working on Human AI Marketing — where we bring big-budget marketing capabilities to Tri-Cities businesses by pairing apprentice talent with AI tools we operate as a team. This handbook is everything you need to start contributing safely.

I.

Welcome

You'll be building and maintaining client sites, writing copy, and generating creative with our AI tools. The work is real — what you ship reaches local business owners in Batavia, Geneva, and St. Charles.

A few things we operate as a single team — not per intern, not per project — and treat as shared production resources:

  • Gemini Nano Banana — Google's image-generation API
  • OpenAI Image 2 — OpenAI's image-generation API
  • Cloudinary — media storage, transformation, and delivery

Because the bills are pooled, leaked credentials are everyone's problem. The rest of this handbook is mostly about protecting that shared trust.

II.

Prerequisites

We work entirely from the terminal — no IDE. Install these before your first day. All are free.

  • A terminal
    • macOS: built-in Terminal app. Find it in Applications → Utilities → Terminal, or press ⌘ + Space and type "Terminal".
    • Windows: use Git Bash, which is installed as part of Git for Windows (next item). It gives you the same bash shell, commands, and keyboard shortcuts we use on Mac. Open it from the Start menu after install.
  • Git
    • macOS: usually pre-installed. Run git --version to confirm. If missing, install Xcode Command Line Tools.
    • Windows: download from git-scm.com/download/win. The installer also gives you Git Bash. Use all default options.
  • Node.js (LTS) — JavaScript runtime. nodejs.org (download for your OS; the installer handles the rest).
  • Claude Code CLI — our AI pair-programmer; this is how we edit files. docs.claude.com/en/docs/claude-code
  • nano — simple terminal text editor. Already there: pre-installed on macOS Terminal, and bundled with Git Bash on Windows. We use it for files you need to edit by hand (like .env), since pasting secret keys into Claude is off-limits.
  • A GitHub account. Send your username to Gary so you can be added to the org and get repo access.
  • GitHub CLI (gh) — lets Claude open PRs, fetch preview URLs, and check PR status from the terminal so you never have to leave your work.
    • macOS: brew install gh, or download from cli.github.com.
    • Windows: winget install --id GitHub.cli in PowerShell, or download from cli.github.com.
    • Then, once: run gh auth login and follow the prompts to link your GitHub account. You only do this one time per machine.

III.

Getting your API keys

Each project repo includes a password-protected file called apikeys.zip in the root. Inside is a single .env file with all three credentials Gary maintains for the team:

  • GEMINI_API_KEY
  • OPENAI_API_KEY
  • CLOUDINARY_URL

To open the zip, you need the unzip password. Gary sends it to you through 1Password (or another secure share) — never in Slack, email, or any public channel. Once you have it, you'll use it in Section V to extract your .env.

Do not

  • Share the unzip password — or any value from the unzipped .env — in Slack, Discord, email, or any public channel.
  • Paste either the password or the keys into a Claude conversation directly. Keep them in .env instead.
  • Commit the unzipped .env to git. (It's gitignored by default — don't fight that.)
  • Paste keys into a browser DevTools console.
  • Share the password or keys with anyone outside the team — including other developers, friends, or family.

If the unzip password or any key is exposed by accident, tell Gary immediately. We rotate keys (and the password) quickly, and the cost of a quiet leak is much higher than the cost of admitting one.

V.

The .env file

Each project includes a password-protected file called apikeys.zip at the root. Open it once on your machine to create your local .env file:

unzip apikeys.zip

When prompted for the password, paste the one Gary sent you. It won't echo as you type — that's normal. On success, a .env file appears in your project root, containing all three credentials and ready to use.

Windows: if unzip isn't available

Most Git Bash installs include unzip. If yours doesn't (you'll get command not found), use File Explorer instead: right-click apikeys.zip → Extract All…, enter the password when prompted, then move the resulting .env file back into your project root.

The .env file is gitignored, so it stays on your machine and never reaches GitHub. The apikeys.zip file itself is committed and shared across the team — the password is what keeps its contents safe.

Editing the .env later

If a key gets rotated and you need to change a value by hand, open .env in nano:

nano .env

First time using nano?

nano is a text editor that lives inside your terminal — there's no new window or app. When you run the command above, your terminal screen transforms into the editor and shows the file's contents. Move around with arrow keys, type to add text, and use the shortcuts in the bottom-row menu to save and exit. The ^ symbol in that menu means the Ctrl key — so ^X means Ctrl + X. When you exit, your terminal returns to normal.

Works the same way on macOS Terminal and Git Bash on Windows.

In nano:

  • Paste — macOS: ⌘ + V. Windows (Git Bash): right-click → Paste, or Shift + Insert.
  • Save — Ctrl + O, then Enter to confirm the filename (both platforms).
  • Exit — Ctrl + X (both platforms).

If Gary updates apikeys.zip after rotating keys, the simpler path is to delete your existing .env and re-run unzip apikeys.zip — the new zip becomes your source of truth.

Can't see the file?

Files starting with a dot (.env, .gitignore) are hidden by default in graphical file browsers.

  • macOS Finder: press ⌘ + Shift + . to toggle hidden files on and off.
  • Windows File Explorer: open the View menu → Show → check Hidden items.
  • Either OS, from the terminal: run ls -la to list all files including hidden ones.

Verify your .env works

Before you start any real work, run the smoke-test script. It confirms all three API keys actually work by generating one image with Gemini, one with OpenAI, and uploading both to Cloudinary. From the project root:

node scripts/test-apis.mjs

If everything is set up correctly, you'll see this at the end:

========================================
  ALL THREE APIS WORK
========================================

  Gemini image:  https://res.cloudinary.com/.../gemini-...png
  OpenAI image:  https://res.cloudinary.com/.../openai-...png

Open the two Cloudinary URLs in your browser to confirm the images actually rendered. If a step fails, the error message names which API rejected your key — fix that value in .env (nano .env, save with Ctrl + O, exit with Ctrl + X) and run the script again.

Cost & cleanup

Each run costs roughly 5 cents in API fees (one Gemini image + one OpenAI image). Don't worry about running it more than once. The test images upload to an env-test/ folder on Cloudinary — Gary can delete them from the Cloudinary console whenever the folder gets noisy.

VI.

Claude Code safety

Each project ships with two safety files that run in the background:

  • .claudeignore — tells Claude Code to never index your .env
  • .claude/settings.json — shared sensitive-pattern rules

You don't need to interact with these. But there are still habits to keep:

  • Don't paste raw keys into Claude conversations. Reference them as $OPENAI_API_KEY if needed; let Claude read .env patterns through your shell.
  • Review every diff before committing. Claude is fast and helpful, but you are the engineer of record. If a generated commit drops in an unfamiliar import, ask why.
  • When Claude needs to run scripts that touch APIs, load your .env values into the terminal before you launch Claude. Run both commands in the same terminal window, in this order:
# Step 1 — load .env into your current shell (do this FIRST)
export $(cat .env | xargs)

# Step 2 — now launch Claude (it inherits the loaded env vars)
claude

If you start Claude first and then try to export, the variables won't reach Claude — you'd have to quit (Ctrl + C) and start over.

VII.

Daily workflow

First: which mode is the project in?

Projects pass through two phases. Which one you're in changes how you ship.

Build mode

Site isn't live yet. No customers reaching it, no client reviewing changes. You (and maybe Gary) are prototyping fast.

Direct push to main is fine. Branches and PRs are overhead you don't need yet.

The flow:

  1. git pull — get the latest
  2. Edit (with Claude)
  3. git commit -m "…"
  4. git push — straight to main

Live mode

Site is in production. Real customers reach it, the client may be reviewing changes, multiple people may touch the code.

Always branch + PR + preview + merge. The 8-step flow below.

The flow:

  1. Pull main
  2. Make a branch
  3. Edit (with Claude)
  4. Commit
  5. Push branch
  6. Open PR (preview URL appears)
  7. Check preview yourself / send to client
  8. Merge to main

How to tell which mode you're in: Is there a public URL where real customers reach this site today? Live mode. Could you break things without anyone outside the team noticing? Build mode. When in doubt, ask Gary before your first push — guessing wrong in either direction has real costs (extra friction in Build mode, broken site for customers in Live mode).

The Live mode workflow, in eight steps

1. Pull the latest main

git checkout main
git pull

git checkout main switches you to the main branch (production code). git pull downloads any new commits from GitHub and merges them into your local copy. Always do these two together, in this order, at the start of each work session — otherwise you'll be building on stale code.

2. Make a branch for your work

git checkout -b feat/<your-name>/<short-description>
# example: feat/maya/testimonial-section

A branch is your own personal copy of the code, isolated from everyone else's work. While you're on your branch, you can change anything without affecting main or what other interns are doing. When the work is reviewed and approved, your branch is merged back in.

Branch even for one-line fixes. Name it feat/your-name/short-description so it's clear at a glance who's working on what.

Why branches matter for interns

If you skip branching and edit main directly, your unfinished work goes straight to production the moment you push — and you'll collide with anyone else editing the same files. Branches give every intern their own sandbox.

3. Edit files (Claude does the heavy lifting)

This is where you describe what you want, and Claude Code makes the changes. Claude also handles the git add step (which tells git which files to include in your next commit), so you don't usually need to run it yourself.

4. Save your work with a commit

git commit -m "Add testimonial section to homepage"

A commit is a save point — a snapshot of your changes recorded in your local git history. It does not reach GitHub yet; it lives only on your laptop. You can make many commits in a row as you work.

Write messages in clear, present-tense words: "Add testimonial section," "Fix mobile nav overlap," not "stuff" or "wip." Your future self and your reviewer will thank you.

5. Push your commits to GitHub

# first time pushing a new branch
git push -u origin <branch-name>

# every push after that
git push

A push uploads your local commits to GitHub. Once pushed, your work is visible to the team in the remote repo. The -u origin <branch-name> on the first push sets up tracking — after that, plain git push knows where to send things.

Commit vs. push, in one sentence

git commit saves a snapshot to your laptop only. git push uploads those snapshots to GitHub, where the team can see them. You can commit many times before pushing; you cannot push without committing first.

6. Open a Pull Request

The easiest way: ask Claude. In your terminal, say something like:

Push my branch and open a PR. Give me the preview URL when Netlify has it.

Claude runs gh pr create behind the scenes, then watches the PR for the Netlify preview comment and hands you the URL once it's live. You never have to leave the terminal.

If you'd rather do it manually: go to the project's page on GitHub, click the "Compare & pull request" banner for your branch, write a sentence or two describing what you changed, and create the PR.

Either way, the moment your PR exists, Netlify automatically builds a preview deploy — a temporary URL where your branch is live, exactly as it would appear in production.

Share previews with clients

The preview URL is a real, working website with just your branch's changes. It's perfect for sending to a client for sign-off before anything goes live. The URL stays active until the PR is merged or closed, and it auto-updates with every new push to the branch — so you can iterate based on client feedback and just send the same link again.

7. Check your own work on the preview

Open the preview URL and click through everything you changed:

  • Does the thing you built actually do what you intended?
  • Does it look right at desktop sizes and at phone-sized widths?
  • Did anything unrelated break elsewhere — links, other pages, layouts you didn't touch?

If you find a problem, fix it locally, commit, and push again to the same branch. The preview auto-rebuilds with every push. Repeat until you're satisfied.

Each person checks their own work

There is no separate reviewer waiting to catch your mistakes. Once you click Merge, your changes are live on the real site. So before you merge: open the preview, click around, confirm the change works and nothing else broke. The bar is the same one Gary holds himself to — ship it only when you'd want to see it live.

If you're genuinely unsure whether something looks right, drop a link to the preview in #interns or DM Gary before you merge — much cheaper than rolling back from production.

8. Merge the PR to main

Once the preview looks right, you can merge from the terminal — just tell Claude:

The preview looks good. Merge the PR and delete the branch.

Claude runs gh pr merge --delete-branch behind the scenes. Netlify auto-deploys main to production within a minute or two, and your branch is cleaned up.

Or, manually: scroll to the bottom of your PR on GitHub, click Merge pull request, then confirm. Click the button to delete the branch when it appears next to the merge confirmation.

Never push directly to main on a Live mode project

On Live mode, main serves real customers right now. Your changes always go through a branch and a PR — even when you're the one merging. The PR is what creates the preview you test against; skip it and you'd be shipping blind. (Build mode is different — see the top of this section.)

A worked example: Live mode in practice

Scenario. Your client, the St. Charles Bakery, has a live site at stcharlesbakery.com. Sarah from the bakery emails: "Can we add a testimonials section to the homepage?"

1. Pull and branch

git checkout main
git pull
git checkout -b feat/maya/add-testimonials

2. Tell Claude what you want

"Add a testimonials section to the homepage. Pull three quotes from the document Sarah sent yesterday. Use the same card pattern as the services section."

Claude edits the files.

3. Commit, push, and get the preview URL — all in one ask

"Commit those changes, push my branch, and open a PR. Give me the preview URL when Netlify has it."

Claude returns the preview URL:

https://feat-maya-add-testimonials--stcharlesbakery.netlify.app

4. Check the preview yourself, then send to Sarah

Open the URL. Confirm the testimonials look right on desktop and at phone width. Click around the rest of the site to make sure nothing else broke. Then email Sarah:

"Hi Sarah — here's the testimonials section we discussed: [preview URL]. Let me know if you'd like any changes."

5. Iterate on Sarah's feedback

Sarah replies: "Looks great, but can the quotes be a bit bigger?"

"Bump the testimonial quote font size to about 1.25rem, commit, and push it."

Claude edits, commits, pushes. The same preview URL auto-updates with the new build. Send Sarah the same link again.

6. Sarah approves — merge it

Sarah: "Perfect, ship it."

"The preview looks good and the client approved. Merge the PR and delete the branch."

Claude merges. stcharlesbakery.com auto-deploys with the new testimonials within a minute or two. Done.

VIII.

Where AI calls happen

This is the single most important rule in this document. Read it twice.

Never call AI APIs from browser JavaScript

If you write fetch("https://api.openai.com/...") in client-side code, anyone visiting the site can open browser DevTools, see the API key in the request, and start running up bills on our shared account.

Instead, every AI call goes through a Netlify Serverless Function, which runs on the server (where the key is safe):

Browser  →  /.netlify/functions/your-function  →  AI API  →  Browser

Function files live in /netlify/functions/. They read keys from process.env. A minimal example:

// netlify/functions/generate-image.js
export default async (req) => {
  const { prompt } = await req.json();

  const res = await fetch("https://api.openai.com/v1/images/generations", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${process.env.OPENAI_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({ model: "dall-e-3", prompt }),
  });

  return new Response(await res.text(), { status: res.status });
};

The browser calls /.netlify/functions/generate-image. The key never leaves the server.

IX.

Deploying

Netlify watches each project's main branch. When a PR merges to main, the site auto-deploys to production within a minute or two.

Every PR also gets a preview deploy — Netlify posts a unique URL as a PR comment. This is where you check your work before merging it to main. See step 7 of the daily workflow.

If the preview build fails, click the "Details" link from the PR to see the error log. Build failures are usually:

  • Missing environment variable (Gary will add it in the Netlify dashboard)
  • Syntax error your editor didn't catch
  • A file path that works on your machine (macOS and Windows both treat filenames as case-insensitive by default) but not on Netlify's Linux build servers (case-sensitive)

XI.

Pre-commit security checklist

Run through this every time before you push. It takes ten seconds and will save you the worst week of your apprenticeship.

  • Run git status — is .env anywhere in the output? If yes, stop. Don't commit.
  • Search the diff for the strings sk-, cloudinary://, and API_KEY=. None should appear in tracked files.
  • Any new fetch() calls — are they in /netlify/functions/, or in client-side code? If client-side and they touch an AI API, move them.
  • No screenshots, recordings, or pasted logs with secret values visible.
  • You ran the site locally and confirmed the change works before pushing.

More on commits

Why multiple commits before a push?

A commit is a focused save point. A push is "going public." Doing several small commits before a single push gives you four real benefits.

1. Undo specific things

If a later commit breaks something, you can revert just that one commit and keep the rest of your work intact. One huge commit forces you to throw out the good with the bad.

2. Readable history

Six months from now, running git blame on any line points to the commit that introduced it — with whatever message you wrote. "Add scroll-fade animation" answers the question instantly. "stuff" doesn't.

3. Easier PR review

Small commits let Gary follow your thinking step by step. One huge commit makes review slow, and bugs slip past unnoticed because there's too much to read at once.

4. Safe experimentation

Commit, try something, commit again. If experiment #3 turns out to be a dead end, you can rewind to commit #2 — without losing the in-between work you might still want.

Rule of thumb

If your commit message has the word "and" in it, it should probably be multiple commits instead:

"Add testimonials and fix nav and update colors" → three separate commits.
