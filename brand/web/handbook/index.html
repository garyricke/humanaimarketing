Setting up the shared API keys.

Three production API services are shared across the team and used on every project at Human AI Marketing. This article covers how you get access to them, set them up locally, use them safely in code, and avoid leaking them.

I.

About the shared services

Three production services are operated as a single team account — not per intern, not per project — and treated as shared production resources:

  • Gemini Nano Banana — Google's image-generation API
  • OpenAI Image 2 — OpenAI's image-generation API
  • Cloudinary — media storage, transformation, and delivery

Because the bills are pooled, leaked credentials are everyone's problem. The rest of this article is mostly about protecting that shared trust.

II.

What you need from Gary

When you clone a project, almost everything you need to use the shared API keys is already in the repo:

  • apikeys.zip — a password-protected archive containing all three credentials
  • scripts/test-apis.mjs — a smoke test that verifies your setup works
  • .gitignore — already configured to keep your unzipped .env out of git
  • .claudeignore and .claude/settings.json — already configured so Claude Code never indexes your secrets

The one thing not in the repo is the unzip password. Gary sends it to you through 1Password (or another secure share) — never in Slack, email, or any public channel. Without that password, the contents of apikeys.zip stay locked even though the file itself is right there in the project.

Once you have the password, you'll use it in Section III.

Do not

  • Share the unzip password — or any value from the unzipped .env — in Slack, Discord, email, or any public channel.
  • Paste either the password or the keys into a Claude conversation directly. Keep them in .env instead.
  • Commit the unzipped .env to git. (It's gitignored by default — don't fight that.)
  • Paste keys into a browser DevTools console.
  • Share the password or keys with anyone outside the team — including other developers, friends, or family.

If the unzip password or any key is exposed by accident, tell Gary immediately. We rotate keys (and the password) quickly, and the cost of a quiet leak is much higher than the cost of admitting one.

III.

Set up your local .env

Each project includes a password-protected file called apikeys.zip at the root. Open it once on your machine to create your local .env file:

unzip apikeys.zip

When prompted for the password, paste the one Gary sent you. It won't echo as you type — that's normal. On success, a .env file appears in your project root, containing all three credentials and ready to use.

Windows: if unzip isn't available

Most Git Bash installs include unzip. If yours doesn't (you'll get command not found), use File Explorer instead: right-click apikeys.zip → Extract All…, enter the password when prompted, then move the resulting .env file back into your project root.

Editing the .env later

If a key gets rotated and you need to change a value by hand, open .env in nano:

nano .env

First time using nano?

nano is a text editor that lives inside your terminal — there's no new window or app. When you run the command above, your terminal screen transforms into the editor and shows the file's contents. Move around with arrow keys, type to add text, and use the shortcuts in the bottom-row menu to save and exit. The ^ symbol in that menu means the Ctrl key — so ^X means Ctrl + X. When you exit, your terminal returns to normal.

Works the same way on macOS Terminal and Git Bash on Windows.

In nano:

  • Paste — macOS: ⌘ + V. Windows (Git Bash): right-click → Paste, or Shift + Insert.
  • Save — Ctrl + O, then Enter to confirm the filename (both platforms).
  • Exit — Ctrl + X (both platforms).

If Gary updates apikeys.zip after rotating keys, the simpler path is to delete your existing .env and re-run unzip apikeys.zip — the new zip becomes your source of truth.

Can't see the file?

Files starting with a dot (.env, .gitignore) are hidden by default in graphical file browsers.

  • macOS Finder: press ⌘ + Shift + . to toggle hidden files on and off.
  • Windows File Explorer: open the View menu → Show → check Hidden items (the setting persists until you uncheck it).
  • Either OS, from the terminal: run ls -la to list all files including hidden ones.
  • macOS, permanently: to make Finder always show hidden files without toggling, run this once in Terminal: defaults write com.apple.finder AppleShowAllFiles -bool true && killall Finder. Reverse later by swapping true for false.

Verify your .env works

Before you start any real work, run the smoke-test script. It confirms all three API keys actually work by generating one image with Gemini, one with OpenAI, and uploading both to Cloudinary. From the project root:

node scripts/test-apis.mjs

If everything is set up correctly, you'll see this at the end:

========================================
  ALL THREE APIS WORK
========================================

  Gemini image:  https://res.cloudinary.com/.../gemini-...png
  OpenAI image:  https://res.cloudinary.com/.../openai-...png

Open the two Cloudinary URLs in your browser to confirm the images actually rendered. If a step fails, the error message names which API rejected your key — fix that value in .env (nano .env, save with Ctrl + O, exit with Ctrl + X) and run the script again.

Cost & cleanup

Each run costs roughly 5 cents in API fees (one Gemini image + one OpenAI image). Don't worry about running it more than once. The test images upload to an env-test/ folder on Cloudinary — Gary can delete them from the Cloudinary console whenever the folder gets noisy.

IV.

Using the keys in a Claude Code session

Most of the day-to-day work — generating placeholder imagery for a landing page, batching transforms, uploading assets — happens inside a Claude Code session, not in production code. There are two ways the keys reach Claude when you're building, and it's worth knowing which is which.

Two paths, two key locations

Path 1 — MCP server tools (one-off prompts). Gary has installed a handful of Model Context Protocol servers globally on your machine, including nano-banana (Gemini image gen) and openai-images (OpenAI image gen). Each server was configured with its API key once, when it was installed — the keys live in Claude's global MCP config, not in this project's .env. From your end it's invisible: you ask, Claude calls the tool, the tool already has the key.

Path 2 — Node scripts Claude writes and runs (repeatable jobs). Anything that needs to run again later, or process more than a few files, is better as a script in scripts/. Those scripts read the project's .env at runtime — the same file you unzipped in Section III. Claude writes the script, runs it via Bash, and the script handles its own key loading.

Why Claude itself never reads .env

The repo's .claude/settings.json blocks Claude's filesystem tools from opening .env directly. A Node script Claude writes can read it — because it runs as a separate process — but the raw key never appears in the conversation transcript. That keeps secrets out of the chat log and out of any future context-window snapshot, while still letting the code Claude writes do real work.

A worked example: placeholder hero images for a landing page

Say you're building a new landing page and you need three hero images while the photographer is still scheduling the shoot. A prompt like this is enough:

Use nano banana to generate three placeholder hero
images for a Tri-Cities coffee-shop landing page —
golden-hour storefronts, warm interiors. Upload each
to Cloudinary under brand/placeholders/, then update
brand/web/index.html so the hero  tags pull from
those Cloudinary URLs, web-optimized.

Behind the scenes, Claude will:

  1. Call mcp__nano-banana__generate_image three times with prompts derived from the brief. The Gemini key is already on the MCP server — you don't pass it.
  2. Save the PNGs locally to generated_imgs/ (which is gitignored — see Section VI).
  3. Write or reuse a small Node script that reads CLOUDINARY_URL from .env, signs an upload request, and pushes each PNG into brand/placeholders/ on Cloudinary with stable public IDs.
  4. Edit brand/web/index.html to point the hero <img> tags at the new Cloudinary URLs, inserting f_auto,q_auto,w_1600 into the transform segment so the browser receives WebP or AVIF at an appropriate width.

The resulting URL has the optimization baked into the path:

https://res.cloudinary.com/dsbllwpbh/image/upload/
  f_auto,q_auto,w_1600/brand/placeholders/hero-1.png

f_auto picks the best format the browser supports (WebP, AVIF, or fall back to PNG/JPEG). q_auto picks a quality level that balances size against perceptible loss. w_1600 caps the width — pair it with a srcset using w_800, w_1200, w_1600 to let the browser fetch only what the viewport needs.

Other prompts that follow this shape

  • "Use openai-images to re-render generated_imgs/hero-batavia.png at golden hour — last 10 minutes before sunset, long warm shadows raking left — then update Cloudinary in place using overwrite=true and invalidate=true so the live page picks it up."
  • "Generate four square avatars with nano-banana for the testimonials section, 512×512, and upload them to brand/testimonials/ on Cloudinary with public IDs avatar-1 through avatar-4."
  • "Look at the current hero image's aspect ratio and write a script that batches three new variations through openai-images at the same crop, uploading each with a variant-N suffix."

Cost & iteration

MCP image calls cost roughly 2–4 cents apiece. Inside a single Claude session you can easily fire a dozen of them while iterating on a prompt — that's fine, but it's not free. Don't leave a loop running, and if Claude is about to regenerate the same image five times to try variations, ask it to commit to one approach first.

V.

Using the keys in production code

Section IV covers the build-time workflow. When an AI call has to happen at runtime — a visitor on the live site triggering image generation, for instance — the rules change. Read this twice.

Never call AI APIs from browser JavaScript

If you write fetch("https://api.openai.com/...") in client-side code, anyone visiting the site can open browser DevTools, see the API key in the request, and start running up bills on our shared account.

Instead, every AI call goes through a Netlify Serverless Function, which runs on the server (where the key is safe):

Browser  →  /.netlify/functions/your-function  →  AI API  →  Browser

Function files live in /netlify/functions/. They read keys from process.env. A minimal example:

// netlify/functions/generate-image.js
export default async (req) => {
  const { prompt } = await req.json();

  const res = await fetch("https://api.openai.com/v1/images/generations", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${process.env.OPENAI_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({ model: "dall-e-3", prompt }),
  });

  return new Response(await res.text(), { status: res.status });
};

The browser calls /.netlify/functions/generate-image. The key never leaves the server.

VI.

Don't leak the keys

Run through this every time before you push. It takes ten seconds and will save you the worst week of your apprenticeship.

  • Run git status — is .env anywhere in the output? If yes, stop. Don't commit.
  • Search the diff for the strings sk-, cloudinary://, and API_KEY=. None should appear in tracked files.
  • Any new fetch() calls — are they in /netlify/functions/, or in client-side code? If client-side and they touch an AI API, move them.
  • No screenshots, recordings, or pasted logs with secret values visible.
  • You ran the site locally and confirmed the change works before pushing.
